### BaseAdapter 当前状态

**代码规模**:
- 总行数：~1300 行
- 方法数量：40+ 个
- 最大方法行数：~150 行

**职责分析**:

| 职责领域 | 方法示例 | 行数估算 |
|---------|---------|---------|
| 错误处理与重试 | `processRequestWithRetry()`, `handleFallback()` | ~200 行 |
| 请求处理模板 | `processRequest()`, `processStreamingRequest()` | ~300 行 |
| 监控集成 | `recordMetrics()`, `updateStats()` | ~100 行 |
| 追踪集成 | `startTracing()`, `endTracing()` | ~150 行 |
| 辅助方法 | `selectInstance()`, `getClientIp()` | ~100 行 |
| 业务方法 | `chat()`, `embedding()`, `rerank()` 等 | ~450 行 |

**问题识别**:

1. ❌ **方法过长**: `processRequest()` 方法超过 150 行
2. ❌ **职责混杂**: 单个方法包含错误处理、重试、监控、追踪等多重逻辑
3. ❌ **难以测试**: 复杂的方法逻辑难以编写全面的单元测试
4. ❌ **维护成本高**: 修改一个功能可能影响其他功能

## 🎯 版本拆分策略

### BaseAdapter 拆分策略

采用**纵向拆分**策略，按功能领域提取为独立的组件类：

```
BaseAdapter (协调层)
    ├── ErrorHandler (错误处理)
    ├── RetryHandler (重试处理)
    ├── RequestProcessor (请求处理)
    ├── MetricsRecorder (监控记录)
    └── TracingManager (追踪管理)
```

**拆分原则**:
1. 保持 BaseAdapter 作为协调层，对外接口不变
2. 每个新组件只负责单一职责
3. 通过依赖注入组装各组件
4. 保持向后兼容，不影响现有 Adapter 实现

### ApiKeyService 拆分策略

采用**横向拆分**策略，按职责提取为独立的服务类：

```
ApiKeyService (协调层/Facade)
    ├── ApiKeyRepository (数据访问)
    ├── ApiKeyValidator (验证器)
    ├── ApiKeyHasher (哈希器)
    ├── ApiKeyRateLimiter (限流器)
    └── ApiKeyAuditLogger (审计日志)
```

**拆分原则**:
1. 保持 ApiKeyService 作为 Facade，对外接口不变
2. 每个新组件可独立测试和复用
3. 通过依赖注入组装各组件
4. 保持事务一致性
